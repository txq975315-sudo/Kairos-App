## 三、面试 24 小时前的速查清单

打印或者截图保存，面试前看一眼：

### 必背
- [ ] 1 分钟自我介绍（`03` 第一部分通用版）
- [ ] 5 道高频问题答案（`03` 第二部分）
- [ ] Q3 STAR 案例：紧急程度颜色实时同步（`03`）
- [ ] Q4 0 用户怎么证明（`03`）—— **这题最容易被问，必背**
- [ ] **定稿口径**（`06` 第一、二节）：纯本地边界、不做清单、Inbox+随手记、小组件只读、每日一句唯一入口、删除无回收站等
- [ ] **差异诚实口径**（`06` 第三节）：若面试官对照代码与文档不一致，如何一句话说清
- [ ] 反向提问 2-3 个（`03` 第四部分）

### 必备
- [ ] 简历 PDF（手机里 + U 盘里 + 邮箱草稿）
- [ ] 作品集 PDF / Notion 链接（手机能 1 秒打开）
- [ ] 项目截图 5-8 张（手机相册置顶）
- [ ] 面试地点 / 视频链接 / 设备测试

### 心态
- [ ] 提前 15 分钟到 / 5 分钟登录
- [ ] 听完问题想 2-3 秒再答，**不抢话**
- [ ] 答案不超过 90 秒，超出让面试官追问
- [ ] 不会的坦白说"这个我没做过，但我会这样想…"，**绝不编**

## 四、最容易被忽视的"差异化加分项"

> 90% 的应届生不会做，做了你立刻和别人不一样：

1. **简历或作品集里附一页「决策记录」链接或摘录**（证明你有 PRD / 需求真源意识，与 `06` 配套）
2. **简历右上角放 Notion 作品集链接**（不是只放 GitHub）
3. **GitHub README 用中英双语**（让海外 HR / 老板也看得懂）
4. **GitHub Topics 加全**（android、jetpack-compose、kotlin、productivity、journaling）
5. **每次面试后 24 小时内发感谢邮件**（用 `04` 的模板）
6. **写一篇深度文章发到小红书 / 即刻 / 公众号**（用 `04` 第八部分模板）
7. **主动提"我准备了一个 1 分钟视频自我介绍"**（如果你愿意录）
8. **在 GitHub 仓库里放一个 `screenshots/` 文件夹**（README 引用，HR 在网页上能直接看到产品长什么样）

## 五、关键原则（贯穿全部材料）

### 原则 1：所有"自夸"必须有"具体证据"

❌ 我有产品思维  → ✅ 我把"日记自由分类"改成"6 个固定主题"，把启动门槛从开放问题变成选择题

❌ 我懂技术 → ✅ 我能看懂 Kotlin 和 SQL，能用 CompositionLocal 这种术语和研发讨论方案

❌ 我能学习 → ✅ 我从"完全不会 AppWidget"到"4 种尺寸上线"用了 4 天

### 原则 2：主动承认局限，永远比"装大佬"加分

→ "目前 Kairos 没有真实用户量级的数据；**个人项目定稿刻意不做商业化埋点**，进团队后我会在合规框架下补数据闭环"（与 `决策记录1`、`06` 第六节一致）

→ "定稿阶段坚持纯本地、无云同步；**Repository 抽象便于未来扩展**，不等于产品承诺立刻上云"

### 原则 3：把"独立"叙事讲到位

应届 0 实习的最大武器不是"我有 X 月经验"，是**"我独立交付过一个完整产品"**——这是 90% 实习生都没机会做的事。

把这件事讲透，你比 80% 同期应届生更有竞争力。

## 六、可能遇到的"为难提问"应对

### Q：「你这个项目就你一个人做，怎么证明你有团队协作能力？」

> 我承认这是这个项目的局限。
> 但我做的过程中刻意搭建了"小团队"协作场景：
> 1. 每周邀请 1-2 位朋友做"15 分钟试用 + 反馈"
> 2. 用固定模板分类反馈（情绪 / 行为 / 期待）
> 3. 反馈出现 ≥3 次才进 backlog
>
> 这件事让我学到：协作的核心不是"人多"，是**信号过滤 + 优先级达成共识**。
> 这两件事我已经在练。

### Q：「你这个项目两个月做完，是不是太赶了？质量能保证吗？」

> 我承认快是事实，但快不等于糙：
> 1. 全程 TDD，44+ 个单元测试覆盖关键路径
> 2. 自定义 Lint 规则 + 硬编码 CJK 扫描，防止技术债
> 3. 6 次完整版本迭代，每次都是闭环：定义 → 实现 → 自测 → 提交
>
> 真正的工程质量不是"做得慢"，是**"过程有控制"**。
> 这一点我用项目证明了。

### Q：「你既会做产品又会写代码，你怎么定位自己？」

> 我对自己的定位很清晰：**我是一个偏产品的候选人，不是开发候选人。**
> 我会写代码，是因为我相信"听不懂研发的语言就做不好产品"，
> 不是因为我想转研发。
>
> 我未来想做的是：用一个产品经理的视角，
> 用一个研发也能听懂的语言，
> 把用户的真实需求翻译成可被执行的产品方案。
