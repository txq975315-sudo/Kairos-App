## 📋 Page 2 · 一句话定位 + 核心数据

```
## 是什么？

Kairos 是一款 Android 端的个人时间管理 App，
区别于市面上"只服务效率"或"只服务记录"的工具，
它把「任务执行」和「生活记录」放进同一个时间空间，
帮助 18-28 岁有自我管理意识但抗拒"硬核生产力"的年轻用户，
找到属于自己的节奏。

## 名字的由来

古希腊有两种时间——
Chronos（钟表时间）：量化的、流逝的、无情的
Kairos（质量时间）：恰当时机，是让生命有意义的瞬间

Kairos 的命题是：让用户既能管理 Chronos，也能捕捉 Kairos。

## 核心数据

🟦 5 个一级模块（Today / Essay / View / Widget / Mine）
🟦 30+ 个 Compose 页面
🟦 4 种尺寸 / 8 种布局的桌面小组件
🟦 34,000+ 行 Kotlin 代码
🟦 6 次完整版本迭代
🟦 中英双语运行时切换
🟦 44+ 个单元测试
```

## 📋 Page 3 · 用户洞察 & 定位

```
## 我观察到的矛盾

我观察到自己和身边同龄人有一组矛盾的需求：

| 一边想要 | 另一边又厌恶 |
|---|---|
| 用 Todo / Notion 提高效率 | 工具太"工作感"，下班后不想再打开 |
| 用日记 / 手帐记录情绪 | 切换 App 太麻烦，记完没人复盘 |
| 看到月历能感受到时间在过 | 但月历里只看到任务，看不到自己的状态 |
| 想要复盘 | 但每周写复盘门槛太高 |

## 核心洞察

年轻用户真正缺的不是"任务管理"或"日记"，
是一个能同时收住理性日程与感性瞬间、
并能在合适的时候把它们联系起来的"个人时间空间"。

## 调研方法

- 12 位 18-28 岁同龄朋友的非结构化访谈
- 自己作为 0 号用户连续使用 4 周以上
- 拆解 Notion / 滴答清单 / 番茄 ToDo / 一言 / 格志日记 等 8 款相关 App
```

## 📋 Page 6 · 两个高难度专项的 Before / After

```
## 专项一：Essay 主题视觉规范重构

### Before
- 6 个主题用 Material 默认色板（饱和度高、缺乏层次）
- 命名"自我认知 / 社会人际 / 存在探索"过于学术化
- UI 层多处硬编码颜色，改一次主题色要动 15+ 个文件

### After
- 升级为「主色 + 辅助色 + 浅背景色」三层视觉系统
- 命名改为"觉察自我 / 社交观察 / 意义探索"+ 邀请性副标题
- 所有 UI 通过 categoryColor() 动态读取，未来再改主题色 0 成本

### 我的工作
1. 重新做主题命名 A/B 比对
2. 重新设计三层视觉系统
3. 用 28 个单元测试锁定行为，再统一替换
4. 改完 6 个文件 + 2 套字符串资源，行为零回归

────────────────────────────────────────

## 专项二：紧急程度颜色实时响应修复

### Before
- 用户改色后已创建任务卡片不变
- 必须杀进程重启 App 才能生效
- 桌面小组件完全不响应

### After
- 用户改色 → App 内任务卡片 + 桌面小组件 < 1 秒同步更新

### 我的工作
1. 定位根因：8 个 Compose 组件直接调用静态变量
2. 设计方案：引入 CompositionLocal 让配置响应式下发
3. 重构 8 个组件 + Widget 同步逻辑
4. 顺手清理一段死代码

> 启发：产品经理在审需求时，不能只验证"主流程能跑通"，
> 要主动追问"用户改完之后，所有相关界面都正确响应了吗？"
```

## 📋 Page 7 · 我的反思 & 下一步

```
## 当前局限（主动承认）

1. 没有真实用户量级的数据反馈，所有判断仅来自自用 + 朋友试用
2. 个人项目定稿刻意不做商业化埋点，定性反馈多、定量漏斗少（进团队后补）
3. 首次冷启动引导弱，新用户面对 5 个 Tab 容易迷茫
4. 协作场景缺失，所有数据都在本地

## 下一步规划（按优先级）

P0 · 新用户 onboarding（3 屏 + 首条任务/笔记引导）
P0 · 代码与《决策记录1》逐项对齐（删除/小组件/入口等）
P1 · View 规则引擎 / 洞察（定稿延后项）
P1 · 备份形态扩展（如 WebDAV，仍可无账号）
P2 · 入职团队后在合规框架内学习埋点与实验（个人定稿刻意不做商业化埋点）
P2 · iOS / 云同步（远期单独立项）

## 我从这个项目里学到的最重要的事

1. 好的产品决策不来自"灵感"，来自对自己每天行为的持续观察
2. 产品经理最难的不是想点子，是在很多好点子里选不做哪些
3. 看不见的细节（实时性、一致性）才是产品力的真正分水岭
4. 0 实习不是劣势——独立交付一个完整 App 是更稀缺的实习
```

## 📖 开发故事

> 这是一个独立完成的产品 + 工程项目，背后有一些值得分享的设计决策。

### 故事 A · 把"日记自由"改成"6 个主题"

最初 Essay 模块允许用户自己创建任意分类，
但自测一周后发现"今天写什么"成了最大的启动门槛。
后来重构为 6 个心理学主题（觉察自我 / 社交观察 / 亲密关系 / 身心状态 / 意义探索 / 自由随想），
把开放性问题变成选择题。

### 故事 B · 一次"实时同步"的细节修复

做完「紧急程度颜色可自定义」功能后发现，
用户改色后已创建任务卡片不变。
通过 CompositionLocal 重构 8 个组件 + 同步小组件刷新，
实现 < 1 秒全端同步。

### 故事 C · 让"每日一句"和"笔记"打通

把"每日一句"从手输文本，改成可从已发布笔记中挑选，
形成"写笔记 → 被自己未来选中"的二次回路，
让两个独立功能变成会持续滚雪球的数据闭环。
