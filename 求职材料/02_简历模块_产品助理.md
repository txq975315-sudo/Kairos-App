## 三点五、项目经历 · 结构化写法（STAR / CAR / PAR / FAB）

> 用途：**简历 bullet** 多用 **FAB / 压缩 STAR**；**面试口述** 用完整 **STAR**；书面作品集可用 **CAR / PAR**。下面均以 **Kairos** 为例，可直接改写人设称谓。

### 1）常用框架对照（一口气认清）

| 框架 | 字母含义 | 最适合 | 简历上一行怎么写 |
|------|-----------|--------|------------------|
| **STAR** | Situation 情境 · Task 任务 · Action 行动 · Result 结果 | 行为面试「讲一件事」 | 通常写 **A+R** 压缩成一句，或标题 + 两行 |
| **CAR** | Context 背景 · Action 行动 · Result 结果 | 比 STAR 短，书面案例 | Context 一行 + Action 两行 + Result 一行 |
| **PAR** | Problem 问题 · Action 行动 · Result 结果 | 强调「解决了什么矛盾」 | Problem 半行 + A + R |
| **FAB** | Feature 做了什么 · Advantage 相对优势 · Benefit 对用户/业务的价值 | **单行 bullet 黄金结构** | `做了什么（F）` + `因而怎样（B）`，A 可并入 F |

**简历 bullet 公式（推荐）**：  
`[动词] + [对象] + [手段/约束 F] → [可量化或可感知的结果 B]`  
例：「划分 5 Tab 与 Essay Inbox 路径，将日记启动摩擦从「选题」降为「先写后归类」。」

| **R** | 交付可安装、可演示的个人产品；沉淀 **191** 个 Kotlin 文件、**34,000+** 行代码、**44+** 条单测；自用与熟人可用性反馈驱动迭代（定性为主，承认缺规模化数据）。 |

### 3）STAR · 拆条（面试追问时专用 · 每条约 45 秒）

从 **`01_项目报告` 第六章** 的 **S1～S5 / A1 / B1** 任选一个展开即可。

#### STAR · S1 · 产品边界（战略）

| 项 | 内容 |
|----|------|
| S | 个人工具极易做成「功能堆砌」或过早上云，团队一人时复杂度不可控。 |
| T | 必须先定义 **能不能做** 再定义 **怎么做**。 |
| A | 定稿 **纯本地**；书面 **不做清单**；双轨公式「秩序 + 记录 + 桌面」；换机靠 **JSON/ZIP** 导出而非虚构路线图。 |
| R | 需求真源清晰，后续迭代有锚点；面试可逐项解释取舍。 |

#### STAR · S2 · Essay 启动摩擦（体验）

| 项 | 内容 |
|----|------|
| S | 强制先选课题再写 → 用户流失到系统备忘录。 |
| T | **10 秒内能落笔**。 |
| A | FAB **默认随手记**；**Inbox** 缓冲；深度再用六课题；跨维度事件 **关联课题**。 |
| R | 把「开放题」改成「先写再归类」，日记类典型痛点可被复述为产品语言。 |

#### STAR · S3 · View 本周（信息架构）

| 项 | 内容 |
|----|------|
| S | 周回顾要看「这天做了什么」和「这天写了什么」，两栏日期一对不齐就废。 |
| T | 在 **不做规则引擎** 的前提下让用户仍能 **看见模式**。 |
| A | 左 Todo 右 Essay，**同步滚动**；三镜头少跳转。 |
| R | 用户能扫出「任务清光但笔记空白」等节律洞见——**先看见再智能**。 |

#### STAR · S4 · Mine 打卡（伦理）

| 项 | 内容 |
|----|------|
| S | KPI 堆一页 → 用户不敢再打开。 |
| T | 激励但不能羞辱；不同人关注点不同。 |
| A | **打卡 / 统计双 Tab**；图标阶梯 **无惩罚 ❌**；**13 选 8** 指标自定义。 |
| R | 「镜子」而非鞭子；体现默认值与分层思维。 |

#### STAR · B1 · 紧急程度多端同色（工程 · 追问再用）

| 项 | 内容 |
|----|------|
| S | 自定义紧急色后列表与小组件颜色不一致，像功能坏了。 |
| T | **设置、列表、Widget** 同一套真相。 |
| A | DataStore 持久化；Compose **响应式**下发；Widget **刷新**链路收口。 |
| R | 改色后 **约 1 秒内**全端一致；证明端到端 Owner，**非战略故事**。 |

### 4）CAR · 书面案例块（作品集 / 问答「简述」）

**Context**  
独立负责 Kairos：融合任务与随笔的本地 Android 应用，目标用户为抗拒「硬核生产力」的年轻自我管理群体。

**Action**  
划定产品边界与五 Tab 架构；设计 Essay Inbox 路径与 View 周对照；落地小组件多尺寸与 i18n 工程化；书面沉淀决策记录。

**Result**  
完成可演示 MVP，34,000+ 行 Kotlin，6 轮迭代；形成可讲述的需求取舍与信息架构案例。

---

### 5）PAR · 矛盾叙事（「你发现什么问题」类题目）

**Problem**  
Todo 与日记在用户生命周期里同时需要，却长期在两个 App 里断裂。

**Action**  
用同一套数据模型与导航（Today / Essay / View）把「执行」与「反思」在 **View** 时间维度上重新对齐；本地存储降低隐私与心智负担。

**Result**  
产品设计可被复述为「一条故事线」，而非功能列表。

### 6）FAB · 简历单行 bullet（复制即用）

每条 = **F（事实）** + **A（怎么做）** + **B（价值）**，简历里常压缩成 **1～2 行**。

```
· 【F+A】划分 5 个一级 Tab 并定稿「待办 + 随笔 + 桌面」双轨定位与「不做清单」，
  【B】避免一人项目范围失控，需求评审可对齐真源文档。
· 【F+A】设计 Essay 默认随手记 + Inbox 与六课题 / 关联课题，
  【B】降低日记启动摩擦，把留存痛点复述为可执行方案。
· 【F+A】实现 View 本周 Todo 与 Essay 左右对照 + 同步滚动，
  【B】周回顾场景下「行动 vs 反思」可同一天对齐，支撑「先看见再智能」节奏。
· 【F+A】Mine 侧拆分打卡日历与数据统计，打卡图标无惩罚性设计 + 13 选 8 指标，
  【B】避免数据羞辱用户，兼顾不同关注维度。
· 【F+A】落地中英运行时切换 + Lint / 扫描防硬编码，
  【B】国际化可持续维护，减少后续改文案返工。
```

### 9）PRD 产出与需求迭代 —— 要写，而且可以对齐你们仓库里的真东西

不少同学习惯只讲「做了功能」，面试官又问不出代码细节。**产品岗要显式写出：需求从哪里来、写在哪、怎么随迭代修订。** Kairos 个人项目里，等价于 PRD 的不是单一 `PRD.docx`，而是一套**可追溯文档 + 版本节奏**。

#### 在你项目里，「PRD」对应什么（面试一句话）

| 材料类型 | 仓库路径（示例） | 在简历里可称作 |
|----------|------------------|----------------|
| **需求定稿 / 决策真源** | `决策记录/Kairos APP 开发决策记录1.md`、`决策记录2.md`、`决策记录3.md` | 「结构化需求文档 / BRD+PRD 骨架」「书面决策记录」 |
| **迭代过程与版本摘要** | `canvas/开发进度记录.md` | 「迭代日志 / Release notes」 |
| **专题方案** | `.trae/documents/*.md`（如 Essay 主题、紧急程度方案） | 「专项 PRD / 技术方案附录」 |
| **代码侧真源** | Git commit message、`APP_VERSION` 心智上的「v1～v6」叙事 | 「变更可追溯」 |

> 口径：**独立项目没有评审会**，所以强调「**自我评审：文档 ↔ 实现 ↔ 自测**」三线对账，这就是你的 **PRD 流程**。

#### 简历可直接粘贴的 bullet（PRD + 迭代）

```
· 输出并持续维护《开发决策记录》（定位 / 做不做清单 / 状态机 / 模块级交互），作为需求真源，
  迭代中与实现、自测对账，避免「口头需求」漂移。
· 按里程碑拆分 6 轮主版本迭代（任务核心 → Essay → View → 小组件 → i18n → 主题与紧急程度等），
  每轮闭环：更新文档 → 开发 → 自测 → 提交；重大变更附专题方案（如 Essay 主题重构计划）。
· 当代码与最新定稿冲突时，以决策记录为准做收敛说明（或标注实验能力），保证作品集叙事可复查。
```

（可按版面删减为 1～2 条。）

#### STAR · 「你怎么写 PRD、怎么迭代需求」（面试 60 秒）

| 项 | 填空（Kairos） |
|----|----------------|
| **S** | 一人项目若没有书面需求，两周后自己会忘了「当初为什么不做云同步」。 |
| **T** | 建立可复查的 **需求真源**，并让 **6 轮迭代**都有「改了哪条决策」的痕迹。 |
| **A** | 维护 `决策记录/` 多份文档：全局定位与不做清单、分模块（Essay/View/Widget/Mine）决策表；重大专题写 `.trae` 级实施计划；进度写在 `canvas/开发进度记录.md`；功能闭环跟 Git 提交。 |
| **R** | 面试可打开仓库逐条讲解；与 Offer 侧对照时能说明「这不是灵感清单，而是**可演进规格**」。 |

#### PAR · PRD 迭代（「需求变更怎么处理」）

- **Problem**：早期设想（如强制选课题、回收站方案）与真实上手使用冲突。  
- **Action**：把新结论写回决策记录（废弃 / 延后项列表）；必要時调整状态机描述；再改实现。  
- **Result**：需求文档体现 **变更历史与取舍**，而不是永远正确的静态 PDF——这正是团队里 PRD 的真实形态。

#### 与第三章「中版 / 长版」怎么合并写

在 **「我的工作」** 下可增加一小条 **第 0 点或并入第 1 点**：

```
0. 需求文档与迭代治理
   · 撰写并迭代《开发决策记录》（模块边界、做/不做、交互与状态机），配套迭代日志与专题方案；
   · 6 轮主版本发布节奏与文档—实现对账。
```
