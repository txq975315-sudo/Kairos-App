讲**产品边界、状态机、交互统一口径**时，以仓库 **`决策记录/`** 三份 Markdown 及 **`求职材料/06_决策记录对齐_求职口径.md`** 为真源。若 APK / GitHub 与定稿有出入，面试用 `06` 第三节「诚实口径」一句话说清，避免被质疑简历造假。

**定稿级摘要**（综合《Kairos APP 开发决策记录1》《决策记录2》《决策记录3》）：

- **一句话定位**：**今日待办秩序 + 今日灵感记录 + 周/月桌面可视化**；理性计划与感性记录双轨极简闭环。
- **底层约束**：**纯本地**，无账号、无云同步、无网络依赖（商业版扩展另议）。
- **明确不做（节选）**：云同步 / 注册登录 / 社交协作 / 复杂标签与高级筛选 / 小组件内「每日一句来源」独立配置入口（定稿已删）等——完整清单见决策记录1 §3.2。
- **每日一句**：**Today 页顶部唯一入口**（定稿收敛多入口）。
- **删除交互**：全 App 统一 **左滑 + Snackbar 约 3 秒撤销**，超时**物理删除**；定稿**无回收站**（历史实现若有，按 `06` 说明）。
- **小组件**：定稿为 **只读展示 + 分区域跳转 App**；4 尺寸、多布局、背景（纯色/图/渐变）与刷新策略可配置；（若代码中存在「桌面打勾」等实验能力，如实归为迭代差异。）
- **Todo 硬规则**：单日 **最多 12 条**；四时段 `ANYTIME/MORNING/AFTERNOON/EVENING`；排序以决策记录2 为准（紧急度升序、未完成优先等）。
- **Essay**：FAB 默认**随手记**、零摩擦速记；随手记 → **Inbox（约 15 天缓冲）** 再归永久杂记；深度课题路径不同；**主分类单选 + 关联课题多选**（决策记录3）；View 场景 **原地展开** 防误改。
- **View**：**今日 / 本周 / 本月** 三时间镜头同页切换；本周 Todo 与 Essay **左右对照、同步滚动**；V1 **不做**模式洞察规则引擎。
- **Mine · 记录**：**打卡日历** 与 **数据统计** 双 Tab；打卡图标体系结合「任务完成 + 是否有随笔」且**无惩罚性图标**；指标 **13 选 8** 可自定义排序。
- **视觉**：**素净融入**、暖棕系去灰感；仅 Mine 心情卡/个人卡保留极轻阴影（决策记录2）。

以下 **第三章** 为便于 HR 阅读的展开叙述；若与上表或决策原文冲突，**以决策原文为准**。

## 一、项目一句话介绍（电梯演讲版）

**Kairos 是一款基于古希腊"质量时间"概念的 Android 个人成长 App，把"今天要做什么（任务）"和"今天发生了什么（情绪/笔记）"放在同一个空间，帮助用户在规律执行与灵感记录之间找到自己的节奏。**

> 名字来源：古希腊有两种时间——
> - **Chronos**：钟表时间，量化的、流逝的、无情的
> - **Kairos**：恰当时机，是让生命有意义的瞬间
>
> 大多数效率类 App 只服务 Chronos（待办、打卡、计时），Kairos 想同时服务 Kairos——把"被记住的瞬间"也变成可被规划、被回顾的资产。

## 二、为什么做这个产品（用户洞察）

### 用户痛点

我观察到自己和身边同龄人有一组矛盾的需求：

| 一边想要 | 另一边又厌恶 |
|---|---|
| 用 Todo / Notion 提高效率 | 工具太"工作感"，下班后不想再打开 |
| 用日记/手帐记录情绪 | 切换 App 太麻烦，记完没人复盘 |
| 看到月历能感受到"时间在过" | 但月历里只看到任务，看不到自己的状态 |
| 想要复盘 | 但每周写复盘门槛太高，写不下去 |

**核心洞察**：年轻用户真正缺的不是"任务管理"或"日记"，是一个**能同时收住理性日程与感性瞬间、并能在合适的时候把它们联系起来的"个人时间空间"**。

### 目标用户画像

- **核心用户**：18–28 岁、有自我管理意识但抗拒"硬核生产力"的学生 / 职场新人
- **使用场景**：早上规划一天 → 中途随手记一句心情 → 睡前看时间轴回顾今天 → 周末做一次轻量复盘
- **核心价值主张**：**「不只是完成多少事，更是过了什么样的一天」**

## 二点五、面试官真正想听的「决策层级」（请先建立这张地图）

> PM 面试里，**战略与边界 > 信息架构 > 体验伦理 > 迭代故事 > 工程细节**。  
> 下面这张表对应「简历 / 自我介绍 / 一面」应优先暴露的内容；**紧急程度多端同步**等在表里一目了然：**有用，但排在后面**。

| 层级 | 面试官在考察什么 | Kairos 上对应什么（示例） | 一面建议 |

| **S · 信息架构** | 复杂信息怎么分区、用户怎么找东西 | **5 Tab**；**任务与笔记分轨**但在 View 重逢；**View 三镜头**；**本周 Todo∥Essay 同步滚动** | **必讲 1 条**：优先 **本周对照（原故事 D）** 或 **5 Tab 分轨** |

| **S · 体验伦理与激励** | 数据会不会羞辱用户、默认值是否善良 | **打卡日历 / 数据统计双 Tab**；图标阶梯**无惩罚性 ❌**；**13 选 8 指标**（原故事 E） | 投 ToC / 社区 / 工具向善岗位时 **强推** |
| **S · 跨模块串联** | 模块是不是孤岛 | **每日一句 ← 笔记**、**Today 唯一入口**（原故事 C） | 讲清「数据回路」即可 |
| **A · 体验与视觉迭代** | 你会不会改、敢不敢推翻自己 | **六主题命名与视觉 + TDD**（原故事 A） | 体现审美与用户语言，**一轮面试讲一个就够** |
| **B · 交付完整性（工程）** | 配置类功能是否「端到端可信」 | **紧急程度自定义 → Compose + Widget 同源刷新**（原故事 B） | **技术向 / 二面 / 被问到一致性时再展开**；勿当作项目唯一高光 |

**自检**：若整场面试只讲了「DataStore + CompositionLocal」，面试官会觉得你有执行力但**缺失产品主轴**——所以材料里把 **S 级放在前、B 级明确降级**。

## 三、产品功能与信息架构

### 五个一级 Tab（底部导航）

```
┌─────────┬──────────┬──────────┬──────────┬─────────┐
│  Today  │  Essay   │   View   │  Widget  │  Mine   │
│ 今日规划 │ 随笔记录 │ 多维视图 │ 桌面组件 │  我的   │
└─────────┴──────────┴──────────┴──────────┴─────────┘
```

### 模块 1 · Today（今日规划）

**核心功能**：用四个时间段（上午 / 下午 / 晚上 / 全天）组织今日任务（定稿四段：`MORNING` / `AFTERNOON` / `EVENING` / `ANYTIME`）

- 任务卡片：标题 + 描述 + 紧急程度（4 级，**空心圆边框色**区分）等；附件与标签能力以当前版本为准
- **每日一句**：顶部展示用户自定义或从笔记引用的句子；**定稿唯一入口在 Today 顶部**（不设多入口分散认知）
- 顶部 TopBar：日期切换、创建任务 / 复盘类入口（若有晨间复盘流程，以决策记录1 状态机为准）
- **排序（定稿）**：紧急度升序 → 未完成优先 → 创建时间降序（与「先急后缓」心智一致）
- **单日上限（定稿）**：**12 条**，超出弹窗拦截，防止任务过载
- **删除（定稿）**：左滑删除 + Snackbar 撤销，无回收站路径
- **重复任务**：决策记录1 将「重复 / 周期任务」列入**不做**清单；若代码仍保留重复能力，面试按 `06` 第三节说明迭代张力
- 多语言与输入法相关能力以当前版本为准

### 模块 2 · Essay（随笔 / 自我觉察笔记）

**这是 Kairos 区别于普通 Todo 的核心模块。**

**关键交互决策（决策记录2）**：点击 FAB **直接进入编辑页**，**默认选中「随手记 / 自由随想」**，顶部可切换课题——**不强制课题弹窗**，降低「不知道写什么」的启动摩擦；随手记保存后进入 **Inbox（约 15 天）** 再归永久杂记，给灵感一个整理窗口。

把深度记录落在 **6 个主课题**（5 个结构化人生课题 + 1 个随手记桶），让用户在需要结构化时再做选择：

| 主题 key（存储） | 叙事用中文名（UI 可能迭代） | 说明 |
|------------------|-----------------------------|------|
| self_awareness 等 | 觉察自我 / 社交观察 / 亲密关系 / 身心状态 / 意义探索 / 自由随想 | 具体色值与文案以 `NoteCardConstants` 与 `strings.xml` 为准；决策记录2 另有「暖棕去灰」课题色表，两套叙述可并存：前者偏品牌化，后者偏早期视觉规范 |

**每条笔记的字段设计（含决策记录3）**：
- **主分类**（单选）+ **二级分类**（行为化 chip，单选）
- **关联课题** `linked_categories`（0～4 个多选；不含 freestyle、不与主分类重复）
- 行为概述 + 正文（主/二级引导句分层展示）
- 场景标签 / 情绪图标 / 关联项目（Project）
- 图片附件（最多 4 张）
- 状态机：`inbox` / `published` 等（以决策记录2 §八为准）

**信息架构延伸**：
- 项目时间轴（Project Timeline）：把同一个项目下的笔记按时间排成一条线
- 主题年度时间轴：按主题查看一年中所有相关的笔记
- 搜索：跨任务 + 笔记的全局搜索

### 模块 3 · View（多视图浏览）

让同样的数据用不同的「时间镜头」看；**V1 只做数据呈现，不做规则引擎**（决策记录2）。

- **今日**：上半 Todo、下半最新笔记等「一日快照」式分区（决策记录2）；笔记支持**原地展开**（决策记录3）
- **本周**：左右两栏 **Todo / Essay** 对照，**日期严格对齐、同步滚动**——让用户一眼看到「同一天做了什么、想了什么」
- **本月**：概览数字 + 课题能量分布等宏观统计（具体图表深度以定稿为准）

### 模块 4 · Widget（桌面小组件）

**工程量与产品价值都高**：4 个尺寸 × **8 种预设布局**（决策记录3）；背景支持纯色 / 本地图 / 渐变，可读性由对比度策略兜底。

| 尺寸 | 布局变体（示例） | 内容侧重 |
|------|------------------|----------|
| 1×1 | 1a / 1b | 极简：日期 + 进度或紧凑 Todo |
| 2×2 | 2a | 日期 + 任务列表 + 每日一句等 |
| 3×1 | 3a～3d | 宽条：多日 + 四时段等变体 |
| 3×3 | 4a 等 | 月历热力 + 任务等 |

- **产品定稿交互**：小组件为 **只读展示**，各区域点击 **跳转 App 内对应页** 完成操作（决策记录2 §六）；刷新策略支持「打开 App 刷新」与固定时间等平衡电量
- **工程实现**：常见为 `AppWidgetProvider` + **RemoteViews**（+ `RemoteViewsService` / `AlarmManager` 等）；若团队文档写过 Glance，可表述为「评估过 Glance，落地选型以兼容性为准」
- 若仓库中另有「桌面打勾」等能力：归为**迭代中与定稿的张力**，面试用 `06` 第三节

### 模块 5 · Mine（我的）

- 个人资料卡、心情日历（拖拽式 Emoji 等交互以决策记录2 为准）
- **全部记录**：拆为 **「打卡日历」+「数据统计」** 双 Tab（决策记录3）；打卡图标结合「任务是否完成 + 是否有随笔」，**无惩罚性图标**
- **统计指标**：**13 个预定义指标中选最多 8 个**展示，支持拖拽排序与卡片管理（决策记录3）
- **课题管理**：Mine 设置内 **六宫格** 进入主课题编辑；**自由随想锁定**，其余主课题可配置名称 / Emoji / 引导句；二级课题有数量与引导词约束
- 设置中心：主题、语言、隐私、小组件、数据导入导出等（**导入支持 JSON/ZIP、合并/覆盖与校验**——决策记录2）
- **删除 / 恢复**：以定稿为准——**物理删除 + 撤销条**，不设回收站；若当前包仍有回收站入口，按 `06` 说明

### 跨模块的"小而美"系统能力

- **运行时多语言**：中英双语；决策记录3 总结为「扫描 → 清单 → strings → 切换 → Lint/CI 防退化」五步
- **数据导入 / 导出**：JSON（及 ZIP）闭环，便于换机与试用数据迁移
- **通知 / 闹钟**：决策记录1 将推送类能力列入**极简边界不做**；若 Manifest 仍含权限，如实说明与定稿的差异
- **首次启动语言选择遮罩**：降低首访成本

## 四、设计决策与权衡（按「面试重要性」重排）

> 口述顺序：**先大后小**。前 6 条为 **S 级（战略 / 架构 / 伦理）**，第 7～8 条为 **A 级（体验迭代 / 桌面战略）**，第 9 条为 **B 级（交付完整性，技术追问再用）**。

### 决策 1（S）· 产品公式、纯本地与「不做清单」——项目的地基

- **公式**：今日待办秩序 + 今日灵感记录 + 周/月/桌面可视化；**理性与感性双轨**，极简闭环（决策记录1）。
- **纯本地**：无账号、无云同步、无网络依赖——优先 **隐私、离线可用、可控复杂度**；换机靠 **JSON/ZIP 导入导出**。
- **不做清单**（面试至少口述 3～5 条）：云同步 / 登录 / 社交协作 / 复杂标签与高级筛选等——完整见决策记录1 §3.2。
- **面试官听什么**：你会不会 **scope**，会不会先定义「不做什么」。

### 决策 2（S）· 为什么任务和笔记是两个 Tab，却在 View 里「重逢」？

- **放弃**：「所有东西都是一种卡片 + tag」——任务与笔记的**状态机与信息密度**完全不同，硬合并双输。
- **采用**：**Today / Essay 分轨**；在 **View** 用今日/周/月镜头把两套数据放在**同一天、同一屏**上对照——**执行与反思在回顾场景对齐**。

### 决策 3（S）· Essay：默认随手记、Inbox、六课题与关联课题

- **摩擦**：打开就要选课题、写得像作文 → 用户流失到系统备忘录。
- **做法**：FAB **直达编辑、默认随手记**；随手记进 **Inbox（约 15 天）**；深度再用 **六课题**；跨维度事件 → **主分类 + 关联课题**（决策记录3）。

### 决策 4（S）· View：三时间镜头、本周左右脑对齐、V1 不做规则引擎

- **三镜头**：今日 / 本周 / 本月；少跳转，统一视察。
- **本周**：左 Todo、右 Essay，**同步滚动、日期对齐**——否则周回顾不成立。
- **克制**：V1 **不做**模式洞察规则引擎——**先看见，再智能**。

### 决策 5（S）· Mine：打卡双 Tab、无惩罚图标、13 选 8

- **结构**：打卡日历 vs 数据统计，感性痕迹与理性指标分房。
- **伦理**：图标阶梯**无惩罚性 ❌**；空白日中性感。
- **差异**：**13 指标自选最多 8 个**，不强迫同一套 KPI。

### 决策 6（S）· 每日一句：从笔记来 + Today 唯一入口

- **回路**：笔记内容反哺 Today；减少「每天硬憋一句话」的空功能。
- **入口**：**Today 顶部唯一**，避免多处修改、认知分裂。

### 决策 7（A）· 六课题与视觉迭代（命名 + 三层色 + TDD）

- **问题**：名称拗口、色彩像调色盘，违背「素净融入」。
- **做法**：面向普通人的命名与副标题；主题色三层体系；大范围改映射前 **TDD**——这是 **体验迭代故事**，适合讲「推翻自己」，但**不顶替**决策 1～6 的主轴。

### 决策 8（A）· 小组件：桌面是第一屏

- **策略**：4 尺寸 × 多布局、背景可读、定稿 **只读 + 跳转 App**。
- **取舍**：不在桌面复刻完整编辑流。

### 决策 9（B · 补充）· 紧急程度可自定义与多端同色——交付完整性，非战略差异化

- **含义**：允许自定义就必须 **列表 / 小组件 / 设置** 同源，否则用户认为功能坏了。
- **实现**：DataStore → 响应式 UI → Widget 刷新；**仅在技术追问或二面展开**，勿当作项目唯一高光。

### S1 · 产品公式与边界（面试官最常缺的「第一句」）

**30 秒版**：Kairos 不是又一个 Todo，而是 **今日秩序 + 今日记录 + 桌面可视化** 三条腿；**纯本地**保证隐私和复杂度可控；**不做清单**里写清了不做云同步、不做登录社交——先定义边界再堆功能。换机靠导出，不上假想的「下周上云」。

**可体现能力**：产品定义、scope、**敢说「不」**。

---

### S2 · Essay：先写起来，再谈归类（随手记 + Inbox）

**背景**：强制先选课题再写 → 用户立刻关掉 App。

**决策**：FAB **默认随手记**；先进 **Inbox** 给缓冲期，再沉淀到课题；真实事件跨心理维度时用 **关联课题**。

**可体现能力**：**降低启动摩擦**、日记类留存问题的标准解法思路。

---

### S3 · View 本周：同一根横线上的「左脑与右脑」（原故事 D · **信息架构首选**）

**背景**：周回顾要看「这天做了什么」和「这天写了什么」——两栏日期对不齐就全废。

**决策**：左 Todo、右 Essay，**同步滚动**；V1 **不做**规则引擎，先让用户**看见模式**。

**可体现能力**：复杂页面结构、时间维度产品、克制。

---

### S4 · Mine：镜子不是鞭子（原故事 E）

**背景**：一张大表堆满 KPI，用户不敢点开第二次。

**决策**：**打卡日历 / 数据统计** 双 Tab；图标阶梯**无惩罚 ❌**；**13 选 8** 尊重个体差异。

**可体现能力**：激励设计伦理、默认值善良。

---

### S5 · 每日一句：吃自家数据（原故事 C）

**背景**：手打每日一句 → 长期空白。

**决策**：从 **已写笔记** 里选；**Today 顶部唯一入口**，模块数据闭环。

**可体现能力**：跨模块产品思维、入口收敛。

---

### A1 · 六课题视觉与命名推翻重来（原故事 A）

**要点**：从「论文化命名 + 调色盘」到普通人能懂的名字 + 三层色 + **TDD 护体**。

**何时讲**：需要展示「我会推翻自己、会和研发一起控回归」时——**不接替 S1～S5 的战略位置**。

---

### B1 · 紧急程度多端同色（原故事 B · **技术追问专用**）

**要点**：自定义颜色后 **Compose 与 Widget** 必须同一真相；DataStore + 响应式下发 + 刷新小组件。

**定位**：**交付完整性**，证明你能端到端收口；**不是**差异化战略故事。

## 七、可改进项 / 下一步规划（面试加分项）

> 主动说出"我做得还不够好的地方"会比"什么都好"更有说服力。

### 当前局限

1. **没有数据可信度验证**：所有功能都是自用 + 朋友试用，没有真实用户量级的数据反馈
2. **缺少定量漏斗**：《决策记录1》将商业化统计埋点列入不做范围，个人项目以定性反馈为主；入职团队后需在合规框架内补数据能力
3. **首次冷启动引导弱**：除了语言选择遮罩，没有 onboarding，新用户面对 5 个 Tab 容易迷茫
4. **协作场景缺失**：所有数据都在本地，未来能不能做"伴侣 / 家人共写"这类轻协作？

### 下一步规划（按优先级）

| 优先级 | 功能 | 预期价值 |
|--------|------|----------|
| P0 | 新用户 onboarding（3 屏 + 首条任务/笔记引导） | 降低 5 Tab 冷启动迷茫 |
| P0 | 与定稿一致的**收尾**：删除/回收站/小组件交互等与《决策记录1》逐条对齐 | 叙事与体验真源一致 |
| P1 | View **规则引擎 / 洞察**（定稿延后项） | 在「只看」跑稳后再智能化 |
| P1 | 备份形态扩展（WebDAV 等） | 仍坚持无账号前提下的迁移焦虑缓解 |
| P2 | （仅商业场景）在隐私与合规框架内学习 **埋点与实验** | 个人项目定稿刻意不做商业化埋点；入职团队后补数据能力 |
| P2 | iOS / 账号云同步 | 远期产品分叉，需单独立项评估 |

## 八、技术栈一览（面试可口述版本）

> 即便面的是产品岗，能讲清楚技术栈会让面试官知道你"懂研发的语言"。

- **平台**：Android（minSdk 29 / target 35）
- **语言**：Kotlin（JVM 17）
- **UI 框架**：Jetpack Compose + Material 3
- **架构**：MVVM + Repository + 单向数据流
- **导航**：Navigation Compose（嵌套 NavHost）
- **持久化**：Room（笔记 / 项目）+ DataStore（设置 / 配置）
- **响应式**：Kotlin Coroutines + StateFlow + CompositionLocal
- **桌面小组件**：AppWidgetProvider + RemoteViews + RemoteViewsService + AlarmManager
- **图片加载**：Coil
- **国际化**：自建 LocalizedStrings（运行时无重启切换）
- **工程化**：自定义 Lint 规则模块 + 硬编码 CJK 扫描 Gradle Task
- **测试**：JUnit 单元测试（TDD 流程）

## 九、Git 提交节奏摘录（说明开发的"产品迭代感"）

不是堆砌 commit，而是**功能闭环式提交**：

```
feat(ui)        : 桌面小组件完成度与布局统一、超大周历格居中、底栏下移
fix(urgency)    : 修复紧急程度颜色修改后已创建任务不响应的问题
feat(essay)     : 重构 Essay 模块六大主题视觉规范
feat(urgency)   : 实现紧急程度自定义功能完整闭环
feat(view)      : 今日 Tab 日期切换对齐周视图 + 时间轴式 Todo/Notes 布局
feat(i18n)      : 全量文案国际化与运行时语言对接
feat(widget)    : 桌面小组件多 Provider、交互与定时刷新
feat(essay)     : Essay 模块 Room 笔记体系、嵌套导航与回收站
```

（若最新定稿已取消回收站，commit 文案属历史痕迹；对外以决策记录为准。）  
每一条 commit 背后都是一个可以单独讲的小故事。
